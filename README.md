A RESTful API for managing devices resources built with **Java 21, spring Boot, PostgresSQL, and Docker**.

The project follows an **API-First approach using OpenAPI 3.0** and supports :
- Pagination
- Filtering
- Partial and full updates
- **Optimistic locking** locking to prevent concurrent update conflicts
- Includes **Unit and Integration tests** with TestContainers

The API contract is defined using an **API-first approach with OpenAPI 3.0** on the file: **openapi/devices-api.yml**

---

## Quick start
Run the application and database using Docker : **docker compose up --build**

The API will be available at : http://localhost:8080/api/v1/actuator/health

Swagger UI : http://localhost:8080/api/v1/swagger-ui/index.html

## Project Overview

This services allows clients to create, retrieve, update, and delete devices resources.

Each device contains the following attributes:

| Field     | Description                                                                 |
|-----------|-----------------------------------------------------------------------------|
| id        | Uniques device identifier                                                   |
| name      | Device name                                                                 |
| brand     | Device manufacturer                                                         |
| state     | Device state (`AVAILABLE`, `IN_USE`, `INACTIVE`)                            |
| createdAt | Device creation timestamp                                                   |
| version   | Version used for optimistic locking to prevent concurrent update conflicts. |

## Architecture

The project follows a **clean layered architecture**.

**Controller**(implementing the endpoints generated by the API-First) → **Service** → **Repository** → **Database**.

Key design decisions:
- API-First approach using OpenAPI
- MapStruct for DTO mapping
- Optimistic locking to handle concurrent updates
- PostgreSQL for persistence
- Docker for containerized deployment

| Layer             | Responsability                         |
|-------------------|----------------------------------------|
| Controller        | REST endpoints implementation          |
| Service           | Business logic                         |
| Repository        | Data persistence using Spring Data JAP |
| Mapper            | DTP <-> Entity mapping using MapStruct |
| Exception Handler | Centralized error handling             |
| Database          | PostgresSQL                            |

## API Design(API-First)

The API contract is defined using **OpenAPI 3.0** on the file openapi/devices-api.yml.

From this specification the API interfaces and Request/Response models are generated automatically.

## Future Improvements

Several improvement could be implemented to future enhance the system and make it production-ready.

### Security
- Add authentication and authorization (Oauth2 / JWT with Spring Security).
- Implement role-based access control for device management operations.

### Performance & Scalability
- Introduce caching (e.g., Redis) for frequently accessed devices.
- Implement database indexing for frequently filtered fields (brand, state).
- Introduce rate limiting to protect the API from excessive requests.

### Resilience & Fault Tolerance
- Implement **Circuit Breaker** patterns (Resilience4j) to protect the service from cascading failures when communicating with external systems.
- Add retry and fallback mechanisms for transient failures.

### Event-Driven Architecture

- Integrate a message broker such as Kafka to publish domain events (e.g., DeviceCreated, DeviceUpdated, DeviceDeleted).

### Observability
- Add distributed tracing (OpenTelemetry / Zipkin).
- Integrate monitoring with Prometheus and Grafana.
- Add application metrics using Micrometer.

### Deployment & Infrastructure
- Add environment-based configuration management.
- CI/CD improvements.
- Deploy containers using Kubernetes for scalability and orchestration.

### Database Management
- Introduce database schema versioning using **Flyway** or **Liquibase** to manage DB migration in a controlled and repeatable database migrations across environments.
